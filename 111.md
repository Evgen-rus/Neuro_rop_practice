Нужно добавить в анализ сделок ROP Assistant новый блок:

`objection_handling`

Цель:
помочь менеджеру заранее подготовиться к вероятным возражениям клиента и отработать их так, чтобы двигать сделку к деньгам: договору, счету, правкам КП, согласованию комплектации, дате решения или честной дисквалификации.

Важно:

* блок нужен пока только для сделок;
* блок должен быть для менеджера;
* стиль мягкий деловой, без токсичности и давления;
* задача не “победить в споре”, а подготовить менеджера к следующему контакту;
* возражения должны выбираться по фактам сделки, а не просто из общего справочника;
* максимум 3–5 вероятных возражений.

## 1. Обновить JSON-контракт сделки

В `analyze_deal.py`, prompt и validation добавить новое поле верхнего уровня:

```json

  "summary": "кратко: какие возражения вероятнее всего и почему",
```

## 2. Правила выбора возражений

Модель должна выбирать возражения по сигналам из истории сделки:

### price / budget

Выбирать, если есть:

* цена выросла;
* клиент говорил про бюджет;
* “дорого”;
* “нет денег”;
* большой чек;
* бюджет не подтвержден.

Цель отработки:
не сразу давать скидку, а выяснить бюджет, критерии сравнения, ценность комплектации и следующий шаг.

### china

Выбирать, если есть:

* Китай;
* китайское решение;
* китайские товарищи;
* ждем вариант из Китая;
* сравниваем с китайским поставщиком.

Цель отработки:
перевести сравнение из “цена против цены” в “цена + комплектация + запуск + сервис + гарантия + обучение + ответственность за результат”.

### competitor

Выбирать, если есть:

* конкурент;
* альтернативный поставщик;
* тендер;
* ждем другое предложение;
* сравниваем варианты.

Цель:
выяснить критерии выбора, кто участвует в сравнении, что у конкурента сильнее, когда решение.

### pause / timing

Выбирать, если:

* клиент взял паузу;
* “вернемся позже”;
* “после отпуска”;
* “до конкретной даты”;
* “пока ждем”.

Цель:
зафиксировать дату возврата, триггер, следующий контакт и ожидаемый результат.

### decision_maker / internal_approval

Выбирать, если:

* неясен ЛПР;
* решение у директора/техдиректора/маркетинга/финансового отдела;
* контакт не принимает финальное решение;
* “надо согласовать”.

Цель:
выяснить участников решения и путь к финальному согласованию.

### technical_doubt

Выбирать, если:

* не хватает технических данных;
* клиент сомневается в решении;
* нужно подтвердить производительность/точность/комплектацию;
* есть вопросы по схеме, таре, этикетке, аппликаторам, линии.

Цель:
не спорить, а запросить конкретные параметры, согласовать критерии технической проверки и следующий шаг.

### quality_risk

Выбирать, если:

* есть сомнения в качестве;
* клиент сравнивает с другим решением;
* важны стабильность, точность, запуск, сервис.

Цель:
показать гарантию, сервис, пусконаладку, обучение, ответственность после внедрения.

## 3. Правила формулировок

`manager_reply` должен быть:

* деловой;
* короткий;
* готовый к использованию;
* без агрессии;
* без “продавливания”;
* без токсичности;
* без фантазий о фактах, которых нет в истории.

Нельзя:

* сразу предлагать скидку;
* спорить с клиентом;
* обесценивать Китай/конкурента;
* писать “у них плохое качество”, если этого нет в фактах;
* давить “покупайте сейчас”;
* использовать плейсхолдеры `{данные}`, “ДОБАВИТЬ”, “todo”.

## 6. Обновить validation.py

Проверить:

* `objection_handling.applicable` boolean;
* `likely_objections` максимум 5;
* `objection_type` только из enum:

  * price
  * budget
  * china
  * competitor
  * pause
  * decision_maker
  * technical_doubt
  * timing
  * quality_risk
  * internal_approval
  * unknown
* `probability`: high|medium|low;
* если `applicable=true`, список не должен быть пустым;
* `manager_reply` не пустой;
* `follow_up_question` не пустой;
* нет плейсхолдеров и служебных слов.

## 7. Обновить prompt

В prompt добавить правила:

* выбери 3–5 наиболее вероятных возражений именно по фактам сделки;
* не перечисляй весь справочник возражений;
* каждое возражение должно помогать менеджеру в следующем контакте;
* цель отработки — продвинуть к деньгам: договор, счет, правки КП, согласование комплектации, дата решения или честная дисквалификация;
* если сделка в паузе, отработка должна быть мягкой: принять паузу, но зафиксировать дату возврата и следующий шаг;
* если есть Китай/конкурент, не обесценивай конкурента, а переводи сравнение в критерии;
* не предлагай скидку первым действием;
* не выдумывай данные о конкуренте, бюджете, ЛПР или сроках.
