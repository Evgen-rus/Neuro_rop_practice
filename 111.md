Нужно усилить анализ сделок ROP Assistant после сравнения с ручным Perplexity-разбором клиента.

Цель:
добавить в JSON-анализ и Markdown-отчет новые управленческие блоки, чтобы отчет показывал не только “что написать клиенту”, но и “как РОПу управлять сделкой, приоритетом и ресурсами”.

Контекст:
на примере deal_18455 наш текущий отчет правильно определил:

* сделка на этапе КП отправлено;
* сумма 8 709 000 руб.;
* клиент поставил проект на паузу;
* клиент ждет альтернативное техническое решение из Китая;
* срок ответа до 10 июля;
* основной контакт — Алексей;
* сделка не продвинулась к счету/договору/оплате.

Но отчет недостаточно явно говорит:

* в каком режиме сейчас вести сделку;
* стоит ли тратить инженерные/технические ресурсы;
* какой жесткий вопрос надо задать клиенту;
* как защищаться от альтернативы/конкурента;
* какой приоритет у сделки для менеджера и РОПа.

Нужно добавить 5 блоков:

1. deal_mode
2. resource_control
3. shaker_question
4. competitor_defense_checklist
5. priority_recommendation

## 2. Правила для deal_mode

Добавить в prompt правила выбора режима:

```text
active_sale:
- клиент вовлечен;
- есть понятный следующий шаг;
- есть движение к КП/счету/договору/оплате;
- бюджет/ЛПР/срок хотя бы частично подтверждены.

managed_pause:
- клиент прямо взял паузу;
- есть дата возврата;
- причина паузы понятна;
- сейчас нельзя активно давить, но нужно держать контроль.

hard_qualification:
- есть крупная сделка, но не подтверждены бюджет, ЛПР, срок или реальность проекта;
- менеджеру нужно задать прямые квалифицирующие вопросы.

nurture:
- потребность есть, но срок покупки далеко;
- сейчас задача — не дожим, а контрольная дата и прогрев.

disqualify:
- клиент нецелевой, бюджета нет, задача не подходит или нет смысла тратить ресурсы.

lost_risk:
- сделка близка к потере: конкурент, тишина, пауза без даты, цена не принята, ЛПР не найден.

unknown:
- в истории недостаточно данных.
```

Для deal_18455 правильный режим должен быть:

```text
managed_pause
```

Причина:
клиент сообщил, что проект на паузе до 10 июля, ждет альтернативное техническое решение из Китая, основной контакт подтвержден.

## 3. Правила для resource_control

Задача блока:
показать РОПу, стоит ли сейчас тратить время инженеров, техотдела, расчетчиков, Инокса, производства.

Примеры логики:

```text
should_spend_engineering_time = false
```

если:

* клиент ждет конкурента/Китай;
* бюджет не подтвержден;
* критерии выбора не зафиксированы;
* ЛПР неизвестен;
* нет конкретного запроса на правку КП;
* сделка в паузе.

```text
should_spend_engineering_time = true
```

если:

* клиент запросил конкретную правку;
* есть срок решения;
* есть шанс перейти к счету/договору;
* техническая доработка прямо влияет на следующий шаг.

Для deal_18455 ожидаемый вывод:

```json
"resource_control": {
  "should_spend_engineering_time": false,
  "reason": "Клиент поставил проект на паузу до 10 июля и ждет альтернативное техническое решение из Китая. До получения критериев сравнения и подтверждения дальнейшего интереса не стоит тратить ресурсы на новые расчеты.",
  "allowed_work": [
    "подготовить аргументы защиты предложения против китайского варианта",
    "поставить контрольное касание на 09-10 июля",
    "подготовить вопросы по критериям выбора"
  ],
  "blocked_work": [
    "делать новые технические расчеты без запроса клиента",
    "повторно подключать Инокс без подтверждения интереса",
    "пересобирать КП без критериев сравнения"
  ]
}
```

## 4. Правила для shaker_question

Добавить в prompt понятие:

Shaker question — это один прямой вопрос, который вскрывает реальное состояние сделки:

* бюджет есть или нет;
* ЛПР найден или нет;
* проект реальный или “изучают рынок”;
* конкурент сильнее или нет;
* срок покупки настоящий или условный.

Примеры:

Для бюджета:

```text
Правильно понимаю, что бюджет на этот проект уже утвержден, или вы пока оцениваете рынок?
```

Для конкурента:

```text
При сравнении с альтернативным решением для вас важнее минимальная цена сейчас или стабильность запуска и сервис после внедрения?
```

Для паузы:

```text
После 10 июля вы будете готовы принять решение по следующему шагу, или это будет только предварительное сравнение вариантов?
```

Для deal_18455 лучше:

```text
При сравнении с китайским решением для вас ключевым будет минимальная цена или надежность запуска, сервис и ответственность поставщика после внедрения?
```

## 5. Правила для competitor_defense_checklist

Блок применим, если в истории есть:

* Китай;
* конкурент;
* альтернативный поставщик;
* тендер;
* “сравниваем варианты”;
* “ждем другое предложение”;
* “дорого”;
* “смотрим аналог”.

Структура:

```json
"competitor_defense_checklist": {
  "applicable": true,
  "competitor_type": "china",
  "defense_points": [
    "сравнить не только цену, но и состав комплектации",
    "уточнить сроки поставки и запуска",
    "проверить гарантию и сервис",
    "обсудить точность и стабильность работы",
    "сравнить ответственность поставщика после монтажа"
  ],
  "questions_to_client": [
    "по каким критериям будете сравнивать наше решение и китайский вариант?",
    "у китайского поставщика уже есть подтвержденная комплектация и сроки запуска?",
    "кто будет отвечать за монтаж, обучение и сервис после запуска?"
  ],
  "risk_if_not_defended": "Клиент может сравнить только цену и выбрать альтернативу, не учитывая риски запуска, сервиса и стабильности работы."

```

Если конкурента нет, ставить:

```json

"competitor_type": "not_applicable",
"defense_points": [],
"questions_to_client": [],
"risk_if_not_defended": "не применимо"
```

## 6. Правила для priority_recommendation

Этот блок нужен, чтобы РОП понимал, сколько внимания сейчас давать сделке.

Варианты priority:

```text
high:
сделка активна, есть деньги/срок/ЛПР, можно двигать к оплате.

medium:
потенциал есть, но есть риски: конкурент, пауза, неясный ЛПР, неясные критерии.

low:
интерес слабый, срок далеко, бюджет не подтвержден.

pause:
есть явная пауза с датой возврата. Не тратить много времени до контрольной даты.

disqualify:
нет бюджета/нецелевой запрос/нет смысла продолжать.
```

Для deal_18455 ожидаемый вывод:

```json
"priority_recommendation": {
  "priority": "pause",
  "reason": "Сделка крупная, но клиент прямо поставил проект на паузу до 10 июля и ждет альтернативное техническое решение из Китая.",
  "time_allocation": "Не тратить значимое время до 09-10 июля. Подготовить короткое контрольное касание и аргументы защиты предложения.",
  "next_review_date": "2026-07-09",
  "what_must_happen_to_raise_priority": "Клиент должен подтвердить, что наше решение остается в сравнении, назвать критерии выбора и согласовать следующий звонок/обсуждение после 10 июля."
}
```

## 9. Обновить prompt analyze_deal.py

В prompt добавить инструкции:

* Определи режим сделки.
* Отдельно оцени, нужно ли тратить ресурсы компании сейчас.
* Если есть конкурент/Китай/альтернатива — сформируй checklist защиты.
* Дай один shaker question.
* Дай приоритет сделки для РОПа.
* Не делай тон токсичным. Вопрос должен быть прямым, но деловым.
* Не выдумывай конкурента, бюджет, ЛПР или дату, если их нет в истории.
* Если данных недостаточно, используй unknown и объясни, чего не хватает.
