# Инструкции для агента

## Перед изменением

1. Прочитай в `ARCHITECTURE.md` разделы **Source of Truth**, **Critical Invariants** и строку для типа задачи в **Where to change code**.
2. Прочитай указанные source-of-truth файлы и связанные тесты. Код и конфигурация важнее документации; о найденном расхождении сообщи в результате.
3. Проверь рабочее дерево. Не перезаписывай и не включай в свою задачу несвязанные пользовательские изменения.

## Неподвижные правила

- Bitrix-интеграция только читает CRM. Не добавляй write-методы Bitrix.
- Не печатай и не коммить `.env`, ключи, webhook URL, персональные CRM-данные, аудио, транскрипты или содержимое `reports/`.
- Все текстовые файлы — UTF-8. Кириллицу не заменяй транслитом или Unicode-escape; при записи JSON используй `ensure_ascii=False`.
- Не обходи change detection прямым запуском LLM-анализаторов без явно запрошенного `--allow-direct-llm`.
- Не создавай Markdown-отчёт до успешной валидации LLM JSON. Правила knowledge/OKF не являются фактами лида или сделки.
- Lead и deal имеют отдельные JSON-контракты, prompts и renderer-правила. При общем изменении проверь второй контур на несовместимость, но не смешивай их поля или поведение.
- Для lead history различай попытку связи, подтверждённый контакт с клиентом и внутреннюю информацию; CRM-активность сама по себе не доказывает содержательный контакт.
- Не запускай Bitrix/OpenAI/production-пайплайны без необходимости задачи и явного подтверждения для платного действия.

## Рабочий процесс

1. Вноси минимальное изменение в существующую точку ответственности. Не дублируй бизнес-логику между CLI, API и frontend.
2. Если изменяется HTTP-contract, согласованно проверь `api/app.py`, `api/jobs.py` при наличии jobs и `frontend/src/api.ts`/`frontend/src/App.tsx`.
3. Если изменяется SQLite, работай через `storage/rop_db.py`; не создавай побочные обращения к таблицам.
4. `README.md` меняй только при изменении быстрого запуска. Детальные операции принадлежат `Docs/*_runbook.md`; `ARCHITECTURE.md` — только при изменении устойчивого архитектурного факта.

## Проверка и завершение

- После изменения Python-логики: `./venv/Scripts/python.exe -m unittest discover -s tests`.
- После изменения frontend: в `frontend/` выполни `npm run lint` и `npm run build`.
- После документации или любого изменения: проверь `git diff --check` и что затронуты только разрешённые файлы.
- Не называй задачу завершённой, пока не проверены затронутый контракт и релевантные команды. Явно укажи проверки, которые не запускались, и причину.
